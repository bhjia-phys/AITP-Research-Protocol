# Requirements: v1.76 Hypothesis Route Re-entry Surface

## Milestone Goal

Close the next post-`v1.75` research-control gap so AITP can turn parked-route
reactivation conditions and follow-up return contracts into an operator-visible
route re-entry surface, instead of forcing operators to infer route return
readiness from separate deferred-buffer and child-topic artifacts.

## Completed Requirements

### Route Re-entry Surface

- [x] `REQ-HREENTRY-01`: runtime surfaces can materialize a bounded re-entry
  summary from deferred-buffer reactivation conditions and follow-up return
  metadata linked to parked hypotheses.
- [x] `REQ-HREENTRY-02`: replay/read paths expose which parked hypotheses are
  waiting, re-entry-ready, or awaiting child-topic return instead of leaving
  re-entry state implicit.

### Coexistence Rule

- [x] `REQ-HREENTRY-03`: the route re-entry surface coexists cleanly with the
  current route_activation surface, deferred-reactivation helpers, and
  follow-up subtopics instead of auto-triggering runtime mutation.

### Verification

- [x] `REQ-VERIFY-01`: the milestone closes with one bounded isolated
  route re-entry acceptance path plus targeted regression coverage for the new
  re-entry surface.
