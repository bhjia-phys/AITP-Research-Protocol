# v3.0 E2E Scientific Assistant Completion Plan

> **Why this milestone exists:**
> v2.9 closed the promotion-review gate materialization gap. The loop can now
> materialize an explicit gate. However, three critical gaps still prevent AITP
> from feeling like a genuinely strong AI research collaborator:
> 1. No real topic has completed a full L0→L4→L2 promotion end-to-end (Jones
>    proof-grade work validates at L4 but stalls before canonical L2 landing).
> 2. The system cannot yet distill a sharp bounded research question from a long
>    source (thesis, paper set, or chapter) — the first step remains too generic.
> 3. There is no complete code-backed research acceptance case where coding is
>    managed as a branch of a research topic rather than an ad-hoc agent workflow.
>
> This milestone closes all three gaps with bounded, replayable proofs.

---

## Target Outcome

- **One real topic completes full L0→L4→L2 promotion** using the Jones 2015
  proof-grade theorem family as the canonical exemplar.
- **One honest negative-result topic completes L0→L4→L2 promotion** to prove
  the L2 surface can host failed conjectures, not just successes.
- **The scRPA thesis topic produces a sharp first bounded question** with a
  concrete novelty target and validation route.
- **One LibRPA/QSGW code-backed topic runs from bootstrap to benchmark** inside
  AITP, with trust state and reusable method outputs stored in the topic graph.
- **All four achievements are replayable** from fresh-topic bootstrap without
  human intervention beyond natural-language steering.

---

## Scope and Protections

### In Scope

- L4→L2 promotion bridge engineering for real topics (not just schema/unit tests).
- `_derive_idea_packet` and first-step shaping improvements for `formal_theory`
  and `code_and_materials` lanes.
- Lane-specific validation route generation.
- One complete code-backed topic acceptance case.
- Replay proofs and regression assets for all three workstreams.
- Minimal HCI polish needed to make replay proofs observable (`aitp status`
  summary mode, `aitp results` view).

### Out of Scope

- General packaging/marketplace work (remains v3.x+).
- Full template library for all task-type × lane combinations (remains v3.x+).
- Cross-paper notation reconciliation at scale (remains v3.x+).
- Mobile/web UI work (OpenWork remains a separate product track).

## Phase 185: Thesis-to-Question Distillation Hardening

**Axis:** Axis 1 (Layer-Internal Optimization)

**Goal:** Improve `_derive_idea_packet` in `orchestrate_topic.py` so that long
sources (thesis, paper set, chapter) produce a genuinely sharp first bounded
research question instead of a generic topic restatement.

### Requirements

- `REQ-DISTILL-01`: The scRPA thesis topic must produce a `novelty_target` that
  is non-empty and specific.
- `REQ-DISTILL-02`: The `first_validation_route` must name a concrete strategy
  (e.g., benchmark-first, symmetry-reduction, or counter-example search) rather
  than "further analysis."
- `REQ-DISTILL-03`: The derived idea packet must expose at least three
  candidate bounded questions when the source is thesis-length.

### Plans

#### Plan `185-01`: Audit Current `_derive_idea_packet` Behavior

**Owner:** analysis
**Files:** `orchestrate_topic.py` (search `_derive_idea_packet`)
**Steps:**
1. Read the current implementation of `_derive_idea_packet`.
2. Run a fresh scRPA thesis bootstrap and capture the generated idea packet.
3. Score the output on three criteria: specificity of novelty target,
  concreteness of validation route, and sharpness of first bounded question.
4. Identify which prompt/template stage causes the dilution.

**Acceptance:**
- Documented scorecard for current scRPA thesis bootstrap.
- Specific line ranges / prompt stages identified as weak points.

#### Plan `185-02`: Implement Multi-Stage Distillation Pipeline

**Owner:** code change
**Files:** `orchestrate_topic.py`, `decide_next_action.py`, schemas
**Steps:**
1. Add a pre-distillation stage: extract the **core tension** from the source
   (what is the central unresolved conflict or choice?).
2. Add a wedge-selection stage: from the core tension, generate 2–4 possible
   **narrow wedges** (small, answerable sub-questions).
3. Add a lane-matching stage: score each wedge against the active lane
   (`formal_theory`, `model_numeric`, `code_and_materials`) and select the
   highest-confidence match.
4. Modify `_derive_idea_packet` to use the selected wedge as the
   `first_bounded_question`, and record the runner-up wedges as `alternatives`.

**Acceptance:**
- Fresh scRPA thesis bootstrap produces a `novelty_target` with ≥10 words of
  specific content.
- `first_validation_route` names one of: `benchmark-first`, `symmetry-reduction`,
  `counter-example-search`, `definition-first-formalization`, or
  `code-reproduction`.
- `alternatives` array contains 2–4 candidate wedges.

#### Plan `185-03`: Add Lane-Specific Distillation Hints

**Owner:** code change
**Files:** `orchestrate_topic.py`, `schemas/`, `tests/`
**Steps:**
1. For `formal_theory`: hints should ask for "the first honest closure target"
   (a definition, a lemma, or a proof obligation).
2. For `model_numeric`: hints should ask for "the first observable or parameter
  verdict that can be checked against a benchmark."
3. For `code_and_materials`: hints should ask for "the first reproducible
  benchmark or validator that can be run against existing code."
4. Inject these hints into the distillation prompt based on the detected lane.

**Acceptance:**
- At least two lanes show measurably sharper idea packets compared to the
  pre-change baseline.
- Tests for lane-specific distillation pass.
